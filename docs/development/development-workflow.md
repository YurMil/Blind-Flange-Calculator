# Development Workflow

This document defines the recommended local development workflow.

## Prerequisites

- Node.js 22 or newer.
- Corepack enabled.
- pnpm 10.33.0, managed through `packageManager` and CI.

Recommended setup:

```bash
corepack enable
corepack pnpm install
```

## Common Commands

Run the standalone app:

```bash
corepack pnpm dev:blind-flange
```

Run TypeScript checks:

```bash
corepack pnpm typecheck:blind-flange
```

Run unit tests:

```bash
corepack pnpm test:blind-flange
```

Build the deployable static app:

```bash
corepack pnpm build:blind-flange
```

Run root aliases:

```bash
corepack pnpm typecheck
corepack pnpm test
corepack pnpm build
```

## Dependency Versions

Pin exact versions in `apps/blind-flange-calculator/package.json`. Do not use `"latest"`. Dependabot (`.github/dependabot.yml`) opens weekly npm update PRs; review lockfile changes carefully and keep CI on `--frozen-lockfile`.

## Change Flow

1. Start from an up-to-date `main` branch.
2. Create a focused branch for one feature, fix, or documentation update.
3. Make scoped changes.
4. Run the relevant local checks.
5. Review generated static output if the build pipeline changed.
6. Commit source and documentation together.
7. Open a pull request or push to `main` only when direct pushes are acceptable for the repository.

## Branch Naming

Recommended patterns:

```text
feature/step-export-improvements
fix/manual-check-pressure-validation
docs/development-guidelines
ci/static-artifact-validation
```

## Commit Messages

Use clear imperative commit messages:

```text
Add static artifact validation to CI
Fix gasket load calculation for custom geometry
Document CAD worker architecture
```

Best practice:

- Keep commits focused.
- Avoid mixing unrelated formatting and behavior changes.
- Mention the domain area when the change touches engineering formulas.

## Code Review Checklist

For every change, review:

- Does the change preserve current calculator behavior unless intentionally changed?
- Are units and standards assumptions explicit?
- Are TypeScript types precise enough?
- Does the UI remain usable inside the static utility frame?
- Does the build still produce `app.html`, `manifest.json`, and `assets/`?
- Is the documentation updated for workflow, deployment, or architecture changes?

## Generated Output Policy

The folder below is generated by the app build:

```text
static/utility-apps/blind-flange-calculator/
```

Best practice:

- Do not edit generated files manually.
- Rebuild with `corepack pnpm build:blind-flange`.
- Commit generated output only when the repository's publication model requires it.

## Dependency Changes

Before adding a dependency:

- Check whether the existing stack already provides the capability.
- Prefer small, maintained, browser-compatible packages.
- Consider bundle size, especially for CAD, PDF, and visualization packages.
- Run a production build and inspect large output changes.

## Documentation Changes

Update documentation when:

- The folder structure changes.
- Build or deployment behavior changes.
- A new test layer is introduced.
- A domain formula or standard interpretation changes.
- A new CAD/export workflow is added.
